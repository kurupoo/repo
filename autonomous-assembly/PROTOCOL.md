# 自律組み立てプロトコル (Autonomous Assembly Protocol)

セッションをまたいで自律的に成果物を組み立てるためのプロトコル。
各セッションは使い捨て(ephemeral)である前提で設計されており、**すべての状態は
このリポジトリ(`STATE.md` / `journal/`)に永続化される**。セッションのメモリに
依存してはならない。

## 原則

1. **リポジトリが唯一の共有メモリ** — セッション開始時に必ず `git pull` し、
   `STATE.md` を読んでから作業する。
2. **1 サイクル = 1 作業単位 (unit)** — 欲張らない。1 サイクルで 1 unit だけ
   進め、コミット・プッシュして終わる。途中で死んでも次のセッションが続きを拾える。
3. **冪等性** — 同じ unit をもう一度実行しても壊れないように作業する。
4. **検証してから進む** — 各サイクルの最後に `checks/verify.sh` を実行し、
   状態と成果物の整合性を確認してからコミットする。

## サイクル手順

各セッション(またはチェックイン)は以下を順に実行する:

1. **同期**: `git fetch origin && git pull origin <branch>` で最新状態を取得。
2. **検証**: `bash autonomous-assembly/checks/verify.sh` を実行。
   失敗したら、まず前セッションの壊れた状態の修復を今サイクルの作業とする。
3. **状態読取**: `STATE.md` を読み、全 unit の状態を把握する。
   - 全 unit が `done` なら **完了** — 最終検証を実行し、PR を ready にして終了。
4. **クレーム(担当宣言)**:
   - `in_progress` の unit が既にある場合:
     - `claimed_at` が **60 分以内** → 他セッションが作業中。何もせず終了。
     - `claimed_at` が **60 分より古い** → 放棄されたとみなし、引き継ぐ。
   - なければ、`todo` かつ依存 (`deps`) がすべて `done` の unit のうち最初のものを選ぶ。
   - 選んだ unit の行を `status:in_progress` に書き換え、`session:` と
     `claimed_at:` (UTC) を記入して **クレームだけを先にコミット・プッシュ**する。
     (プッシュが reject されたら他セッションと競合している — pull してやり直す)
5. **作業**: unit の内容を実施する。成果物は `build/` 配下に置く。
6. **検証**: `bash autonomous-assembly/checks/verify.sh` を実行。
7. **状態更新**:
   - 成功 → unit を `status:done` に更新。
   - 失敗 → `status:failed` に更新し、原因を journal に記録
     (次のセッションが修復から始められるように)。
8. **記録**: `journal/` に `NNN-<unit>-<result>.md` を追記(append-only なので
   コンフリクトしない)。何をやったか・次に何をすべきかを書く。
9. **コミット & プッシュ**: 変更をすべてコミットし、作業ブランチへプッシュする。

## 停止条件

- 全 unit が `done` になり、最終検証が通ったら終了(それ以上サイクルを回さない)。
- 同じ unit が **2 回連続で `failed`** になったら自動継続を停止し、人間に報告する。
- 人間が PR 上で「stop」とコメントしたら即座に停止する。

## セッションの起動方法(トリガー)

`TRIGGERS.md` を参照。send_later による自己チェックイン、Routine(cron)、
